The Hartland Demo

A category manager types a plain-language question into one box. The platform classifies it, routes it to the right agent, compiles it against a governed model of the warehouse, runs it on real data, and answers in sentences and charts — every block showing its work. The thesis it exists to prove: the LLM handles intent only; everything from intent to data execution is deterministic and governed. The seeded plot is an operational incident — a distribution centre that quietly melts down for ~17 weeks — buried in otherwise-flat data, so an AI-driven investigation lands on the true cause while every wrong hypothesis dies on real numbers. It runs bilingually: the same arc in English over hartland_us and Czech over hartland_cz.

Hartland Stores is a fictional Tennessee retailer (HQ Nashville, ≈ $2.1 B/yr across three channels — Stores, Web, and a third-party Marketplace it fulfils from five distribution centres). This is the reference for the whole demo: the data, the model, the fixtures, how to stand the cluster up, and the story plus the satellite demos that hang off it. It draws on the two authoritative field docs it sits beside — demo-transcript.md (the narrative — "the demo is its transcript") and demo-quirks.md (what actually works, and why).

01

Data overview

The underlying data is TPC-DS at Scale Factor 1 (tpc-ds-1g, dsdgen-generated). Both worlds are restored from one pristine dump and put through a deterministic, idempotent pipeline: redate → localize-cz → catalog → seed → recon. Every stage is hash- or snapshot-guarded, so any world is rebuildable from tpc-ds-1g plus the committed scripts. The redate step re-points every date +23 years (1998–2003 → 2021–2026) with believable ages.

Everything else is identical, which is the point (see the catalog delta below). Post-catalog the 18,000-item population carries 10 categories, 42 brands, 99 classes.

The two worlds — hartland_us (USD) vs hartland_cz (CZK)

Two schema-identical Postgres databases on one CNPG cluster (one CNPG, two DBs), one TTR-M model set, two connections (pg-hartland-us / pg-hartland-cz). The physical schema is byte-identical; only the lexicon (en vs cs) and the currency data differ — md.Money is a currency-agnostic decimal domain, so USD vs CZK is a data fact plus a display label, not a structural difference.

  • FX constant = 23. CZ derives from US by FX-scaling every monetary column ×23, rounded to a believable price point. Applied once, guarded by _localize_meta (a second run would compound to ×529 and is refused).
  • Full re-grounding, not a re-skin: CZK currency; geography mapped US-state → Czech kraj distribution-preservingly (stores → Praha, Brno, Ostrava, Plzeň, Olomouc, Liberec); the meltdown DC re-pointed from Memphis to Brno; Czech lexicon, reasons, and catalog.
  • Story mirrored, not re-invented: seed magnitudes are tuned once on US and inherited verbatim by CZ; the S2 deletion is currency-invariant (hash-keyed on cs_order_number), so both worlds delete the same lines. One locale per delivery — never both on stage.

The catalog delta — real names on top of dsdgen gibberish

A full per-item bilingual catalog for all 18,000 items — real product names, brands, manufacturers, sizes, containers per world — so the Product hierarchy (Category → Class → Brand → Manufacturer → Item) is a genuine drill. The generator is deterministic (every choice is sha256(i_item_sk:salt) mod n, no random()).

Why it matters

The catalog rewrite never touches i_item_sk, i_category, price, or the SCD window — the invariants the item-agnostic seeds and recon depend on. Pre- vs post-catalog aggregates are identical except where the seed later moved them. "The catalog is orthogonal to the sales story" is made physically true, not asserted.

02

The model — the Hartland TTR-M

A single TTR-M model set (every file declares package hartland, flattened), a curated TPC-DS subset dressed as Hartland Stores, in five layers plus bindings.

The layer cake

db/ — physical (schema dbo). The TPC-DS tables actually present: the sale/return facts + inventory, plus the dimension tables. Money is decimal(10,2). No profit path exists at all — every net_profit / cost / discount column is deliberately unmodelled (D-6a).
er/ — 19 curated logical entities, prefixes stripped, each with a bilingual displayLabel (e.g. catalog_sales → Marketplace Orders / Objednávky z tržiště). def area hartland is what the agent's Ariadne resolves.
md/ — the ROLAP star: measures (revenue, returnAmount, semi-additive onHandQty…) and dimensions incl. Product with the 5-level drill and DistributionCentre / ReturnReason value-labels.
lexicon/en + cs · binding/ — the same for: targets in both locales (one model, two locales); descriptive attributes bind to physical columns via table-backed def map.

How the model is served & the two rules that bite

veles polls the model from git Collite/hartland@demo-p2 and serves it as a ModelBundle; golem resolves the area via Ariadne and loads the 15 patterns as its preferred_queries.

The two-refresh rule

After any model edit you must refresh both: veles (VelesService/Refresh {force:true} → confirm queries flip to PARSED) and golem (kubectl rollout restart deployment/golem-hartland and -finance) — golem caches the ModelBundle at boot. Skip the golem restart and it keeps sending stale SQL that fails at the translator — which golem masks as STATUS_DONE, 0 rows.

The meta-quirk — never trust a 0-row turn

A failed query node reports STATUS_DONE, rowCount=0, not an error. "0 rows" usually means rejected (translator, empty, or RLS-zeroed), not empty data. The real cause is only in the ttr-query log by correlation_id. Calcite is also stricter than Postgres: a CTE named returns is reserved (→ rets), GROUP BY on a SELECT alias is forbidden, and named params must be single-curly {name} only.

03

Test fixtures

The two Phase-1 worlds become the standard warehouse for all e2e / integration / nightly tests on every cluster (bp-dsk, collite-o1, hartland) — one platform component authored once in olymp (platform/data/hartland-pg/) and composed into each cluster's data tier. Tests default to CZ; EN where language-agnostic. This replaces the old per-context ad-hoc data.

  • Source of truth: the versioned dumps in tpcds-staging/hartland/{us,cz}/ (SeaweedFS), pinned in the restore Job from dump-manifest.md. Re-freeze R0 → one values bump + a Job re-run per cluster.
  • Composition: base/ is a CNPG Cluster (roles hartland + two *_readonly) + two Databases; overlays/<cluster>/ adds the three role-password ExternalSecrets.
Restore is by hand — deliberately

The hartland-restore Job is not in any kustomization, so ArgoCD never re-runs the ~700 MB restore on a sync. Apply it manually after the dumps are staged to that cluster's SeaweedFS (SeaweedFS is per-cluster — a new cluster's bucket must be seeded first).

kubectl --context <c> -n data delete job hartland-restore --ignore-not-found
kubectl --context <c> -n data apply  -f platform/data/hartland-pg/base/restore-job.yaml

It pulls each dump from SeaweedFS, pg_restore --clean --if-exists --no-owner into each DB, GRANTs to the read-only role, and prints a row-count sanity check (item 18000, store_sales 2880404, catalog_sales 1424874). Test contexts read via pg-hartland-{us,cz}, whose namespaceSelectors match harness run namespaces; the olymp re-plan moves tpcds-query → hartland-query, golem-erp → golem-hartland, and repoints the nightly at hartland_cz.

04

Standing the cluster up

Standalone ArgoCD per cluster: bootstrap → root-app → app-of-apps → three ApplicationSets. appset-ops renders one app per platform/* area (auth, data, gateway, monitoring); appset-apps one per service; appset-golems one golem per enabled Shem.

The runtime constellation for a single demo turn

golem-* (kantheon :7420)
  ├─▶ veles.ttr-server :7261 — ResolveArea / model
  ├─▶ llm-gateway.ttr-server :7280 — Bearer ttrk-golem
  └─▶ query-mcp :7307/mcp ─▶ theseus ─▶ translate / validate / worker ─▶ hartland-pg (ns data)

Cross-namespace hosts must be FQDNs. The golems' Shem bundles come from Collite/hartland@demo-p2; the image is pinned (ghcr.io/collite/golem:0.9.4). All secrets pull from a separate vault — hartland.vault.azure.net (Collite tenant): veles-github-pat, ttrk-golem, azure-openai-key, the pg-hartland RO creds, and the ArgoCD repo cred.

Helm extraEnv replaces (re-list the chart's veles env or the validator loses it) · VALIDATE_DEFAULT_TOP_N=100 is set on hartland (see §5) · images must be multi-arch (the node is amd64 — prefix jib with CI=true).

05

The demo story

The transcript is the design artifact — the demo is its transcript. Six beats plus two in-script satellites. Setting: a Monday morning, mid-January 2026, "we just closed 2025."

beatwhat happens
1 · Cold openIris' inbox holds an overnight "Monday channel health brief" (a Hebe routine). Stores flat, Web noise, Marketplace −6.18% FY / −8.65% H2, incident window −10.62% — rendered live from patterns #2 and #1. "Nothing on this stage is a mock-up."
2 · Q&A with Golem (the core)Five turns narrow it: Web is noise-level; the category drop is spread evenly (a product problem hits products — this hit a channel); AOV via plan source free_sql ("not a decision tree, an analyst"); and the pivotal model gap — "what did it cost us in profit?" → it says it doesn't know (profit isn't modelled).
3 · Pythia RCA (centerpiece)A live hypothesis tree with a budget meter. Customer-shift, demographic, promo, and regional branches die on flat numbers; the fulfilment branch survives — reason skew to 39.10%, Memphis −58.99%, a 17-week zero-on-hand streak. Conclusion: Memphis DC stopped fulfilling for ~17 weeks; H2 impact −$44.9M, plus honest loose ends.
4 · Pin itThe conclusion + chart pin to a "Channel Health" dashboard — "the chart re-executes on refresh (a living view); the evidence is frozen (a record). Replay versus reproduce."
5 · Forecast + what-ifPer-channel forecast with confidence intervals (Metis), then a live what-if simulation — a runtime answer, not an R0-frozen fact.
6 · Close the loop"Set this up as my Monday brief" → Hebe creates the routine from chat — the same routine that wrote Beat 1.

The 4-bug cascade behind "72 rows"

The headline turn only works once four independent fixes line up: single-curly {param} (parse) · correct source_language=sql + source_schema=DB (not translator-rejected) · lowercase channel token (filter matches instead of 0 rows) · VALIDATE_DEFAULT_TOP_N=100 (all 72 rows survive the governance cap). Each otherwise fails silently as a phantom "0 rows," and each needs the two-refresh to take effect. The 72 rows are 24 months × 3 channels.

Show the data — raw vs governed

The interesting satellite is the divergence, not the match. Re-run the free_sql AOV answer raw in psql — same numbers. Then show where the governed path deliberately differs: the validator's TopN LIMIT is injected (governance made concrete); Calcite rejects queries Postgres accepts (contract-vs-trust); and RLS scopes the CFO's rows where raw psql sees everything (row-level security doing its job).
